# ADR 0002: Math grading uses a computer algebra system, not string matching

Status: Accepted
Date: Phase 0

## Context

The core promise of a math platform is that it grades math correctly. A student who enters `1/2`, `0.5`, `2/4`, or `sin(pi/6)` for the same answer is right every time. A student who enters `2(x+1)` when the key says `2x+2` is right. A student who writes `x^2 - 1` when the key says `(x-1)(x+1)` is right. String matching fails all of these, and every workaround built on top of string matching (canonical forms authored by hand, long lists of accepted alternates) is fragile and does not scale across an item bank.

Grading also runs untrusted input. The grader parses and evaluates strings that students type. That is a code-execution risk if handled carelessly, and it is a denial-of-service risk if a crafted expression can make evaluation run without bound.

We also want partial credit that reflects mathematical reasoning (step credit) and items that can be regenerated with different numbers each time a student sees them (parameterized items), both of which need a symbolic engine, not a matcher.

## Decision

Math grading uses SymPy for symbolic equivalence, backed by four capabilities and run inside sandboxed workers.

1. **Symbolic equivalence, not string match.** Two responses are equivalent when SymPy determines they are mathematically equal (for example, by simplifying the difference to zero or comparing canonical forms as appropriate to the item type), not when their strings match. This is the default grading mode.

2. **Numeric tolerance.** For numeric answers, comparison uses an explicit absolute or relative tolerance defined on the item, so `0.333` matches `1/3` within the item's stated tolerance rather than by accident.

3. **Step credit.** Items can define intermediate checkpoints so a response earns partial credit for correct steps even when the final answer is wrong. Each checkpoint is a symbolic check, consistent with the main grading path.

4. **Parameterized items.** An item template carries symbolic parameters. Concrete instances are generated by binding those parameters, so the same skill can be practiced repeatedly with fresh numbers, and the correct answer is derived symbolically per instance rather than hand-authored.

**Safety.** All grading runs in sandboxed Celery workers with strict wall-clock timeouts and no `eval`.

- User math strings are parsed with `sympy.parsing.sympy_parser.parse_expr` using a restricted transformation set and a local symbol table only. Python builtins are not reachable from the parse. `eval` and `exec` are never called on user input. This is implemented in `packages/math_core` (`_safe.py`) and covered by tests.
- Every evaluation is wrapped with a wall-clock timeout. Workers add process isolation and per-task time limits on top of the in-process guard, so a pathological expression is bounded by the worker even where the in-process signal guard cannot run.

## Consequences

**Positive**
- Correct grading across equivalent forms, which is the product's core promise, without per-item alternate lists.
- Partial credit and regenerating items fall out of the same symbolic engine rather than needing separate machinery.
- A defined, tested safety boundary for untrusted input: no `eval`, restricted parsing, bounded execution, process isolation.

**Negative and mitigations**
- Symbolic evaluation is heavier than a string compare. Mitigation: grading is off the request path in Celery workers, with timeouts and, where useful, caching of graded outcomes for identical responses to identical items.
- SymPy can be slow or fail to prove equivalence for some hard expressions. Mitigation: per-item grading configuration (equivalence mode, numeric tolerance, timeout) lets authors pick the right check for the item, with a bounded fallback rather than a hang.
- Sandboxing adds operational surface (worker pool, isolation, limits). Mitigation: this is the price of running untrusted math safely and is treated as a first-class part of the grading module, not an afterthought.
